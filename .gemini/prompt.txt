## Testing Protocol (Pragmatic Test-First)

Write tests alongside your implementation. You are NOT required to follow strict
TDD (red-green-refactor) for every change, but you MUST:

1. **New behavior = new tests.** If you add a function, endpoint, or significant
   code path, write at least one test that exercises it. No exceptions.
2. **Bug fixes = regression tests.** If the issue is a bug, write a test that
   reproduces the bug FIRST (confirm it fails), then fix the code so it passes.
3. **Edge cases from the spec.** If the spec or issue lists edge cases or failure
   modes, write tests for those specifically — don't just test the happy path.
4. **Run the full suite.** Before creating your PR, run `uv run pytest -v` and
   `uv run ruff check .`. Fix any failures.

### Coverage Audit (BEFORE writing tests)
Before adding ANY new tests, run a quick coverage audit:
1. List existing test files in the area you're changing
2. Check if tests already exist for the code you're modifying
3. If existing tests cover your change, UPDATE them — don't create duplicates
4. If a test file has placeholder tests (`expect(true, isTrue)` or `skip: true`),
   convert those into REAL tests for the behavior you're implementing — don't add
   new files alongside broken placeholders
5. Focus test effort where coverage is WEAKEST, not where it's already strong

### What NOT to do:
- Don't write trivial tests that assert True or mock everything away.
- Don't skip tests because "the change is small" — small changes cause big regressions.
- Don't add tests for code you didn't write or change (focus on your diff).
- Don't create new test files when existing ones cover the same module — extend them.

# Cross-Repo Mission Delivery Patterns

### 1. Artifact handoff must be explicit

- **Critic reports need an HQ copy**: satellite-local reports are invisible to HQ sign-off flows unless they are copied back with the canonical filename.
- **Manifest paths should stay repo-relative**: stored paths must be valid from the repo that evaluates them, even when the mission spans multiple repos.
- **Sign-off readiness depends on all referenced artifacts**: a correct report path is not enough if the content or naming convention is stale.

### 2. Synchronize specs and work order

- **Satellite specs are inputs, not the source of truth**: when work crosses repos, the manifest and HQ brain pages must describe the same mission state.
- **Cross-repo work needs local validation before handoff**: each repo should verify its own tests and generated artifacts before the mission advances.
- **Mission sequencing should respect dependency order**: shared infra or review artifacts should land before downstream repos try to consume them.

### 3. Delivery discipline

- **Branch protection is part of the integration plan**: admin merges, branch filters, and mergeability checks are delivery constraints, not postscript details.
- **Use the same naming convention end-to-end**: canonical filenames make dashboard automation and human review line up.
- **If one repo stalls, preserve state and continue from the pushed artifact**: continuation tasks should inherit explicit context, not rely on memory.

### jules force-push hazards (2026-05-03)

- **Observed failure mode:** jules branch-update force-pushes can reintroduce older copies of files that were already fixed on `main`, silently widening PR scope.
- **Mitigation:** before merging any Jules-authored PR, diff the current PR against `origin/main` and block if any file falls outside the PR's stated scope; if needed, restore the out-of-scope paths from `origin/main` before the next push.
- **Scope sources:** treat the PR/issue "Files touched" section and linked spec paths as the canonical allowlist. If no explicit scope is present, require XO review instead of guessing.

## Issue Details (pre-fetched)
**Title**: POST / notification path discards metadata — structured questions_batch_response payload unreachable to agent
**Labels**: None

**Description**:
## Problem

The phone-side response to a `questions_batch_request` (and to choice / future structured responses) goes via HTTP `POST /` with body shape:

\`\`\`json
{
  "message": "Submitted 3 answers",
  "metadata": {
    "type": "questions_batch_response",
    "batch_id": "<request_id>",
    "batch_answer": {"answers": [...], "cancelled": false}
  }
}
\`\`\`

But the `/` handler in \`src/http_bridge.ts:354\` reads only \`body.message\` / \`body.content\` and forwards that to \`sendChannelNotification\`. The \`metadata\` field is dropped on the floor. As a result, the agent sees \`"Submitted 3 answers"\` as plain text but cannot read the structured \`{answers, cancelled}\` payload from the inbound \`<channel source="hitl-channel">\` envelope.

## Why now

PR #18 (SPEC-AW-311 fire-and-forget) shipped the dispatch-side fix. The synchronous \`{answers, cancelled}\` JSON return value is gone, so callers were expected to read the structured payload from the asynchronous notification — which is currently impossible.

## Proposed fix

Extend the \`POST /\` handler to forward the \`metadata\` field on the channel notification frame. Agent then sees the inbound message as:

\`\`\`xml
<channel source="hitl-channel"
         message_id="..."
         sender_id="..."
         type="questions_batch_response"
         request_id="..."
         batch_answer="{...}">
  Submitted 3 answers
</channel>
\`\`\`
